One of the first things any Rails book or tutorial will tell you is to keep controllers skinny and models fat. It's nearly as iconic as 'convention over configuration.'

The idea is simple: controllers handle HTTP. Models hold the business logic.

But many of our controllers have a different opinion

app/controllers/app/api/candidates_controller.rb — #tag 
def tag
  authorize candidate
  tags_before = candidate.tags.order(:name).map { |t| {name: t.name, color: t.bg_color} }
  candidate.update(tag_list: params[:tags])
  tags_after  = candidate.tags.order(:name).map { |t| {name: t.name, color: t.bg_color} }

  if tags_before != tags_after
    candidate.actions.create(code: "tags", user: current_user,
      data: {from: tags_before, to: tags_after})
  end

  PusherChannel.candidate_channel(candidate).trigger_async("updated-candidate",
    {candidate: {id: candidate.id, tag_ids: candidate.tag_ids}},
    socket_id: RequestStore[:pusher_socket_id])

  head :ok
end

Diffing tags, logging the activity, broadcasting the change — all domain behavior, all in the controller.

Here's a real action from our codebase. It's not the worst offender — the main candidates and jobs controllers are full of these — but it's small enough to fit on a slide.

Look at what's happening: we're computing a diff of tags before and after, deciding whether the change is meaningful, recording an activity, and broadcasting a Pusher event. Every one of those is something the domain should know how to do.

This isn't a critique of any particular author — it's an observation that we don't seem to share a strong convention here. I'd like to use this as a chance to discuss what we think belongs where, and which direction we want to move in.

Chapter 1

Single Responsibility

What is each layer actually for?

← →

Two orchestrators, one domain

HTTP

Controllers

Orchestrate requests.

def update
  authorize candidate
  candidate.update(...)
  respond_with_json candidate
end

ASYNC

Jobs

Orchestrate async execution.

def perform(candidate)
  candidate.publish
end

↓   Both delegate to the same domain.   ↓
Neither holds business logic.

My view, framed by the single responsibility principle.

Controllers orchestrate HTTP requests. Jobs orchestrate executing something asynchronously. That's their single responsibility.

They are not the place for domain behavior — they wire it together.

Chapter 2

The Standard Shape

Optimize for pattern matching, not cleverness

← →

What every action should do

  • Authentication — ApplicationController concern
  • ID → instance lookup — shared Scoped concern
  • Authorization — single Pundit call
  • Model execution
  • Response — conditional on the result of step 4

When every action follows this shape, reading controllers becomes pattern matching.

You don't read the body line by line — you look at the shape and conclude 'the standard is happening here.'

If something deviates, that's where you focus.

Three lines of body

app/controllers/app/api/candidates_controller.rb 
class App::Api::CandidatesController < App::Api::BaseController
  include CandidateScoped

  def update
    authorize candidate
    candidate.update(permitted_attributes(candidate))
    respond_with_json candidate
  end
end

respond_with_json already handles both branches — serialized record on success, 422 with errors when invalid. The if/else is unnecessary.

Authentication is handled in BaseController. CandidateScoped finds the candidate. authorize is the Pundit call. Then update and respond.

One thing worth pointing out — this isn't vanilla Rails. We don't render manually. respond_with_json is our concern that wraps JsonResponse: it serializes the record on success, and returns 422 with the errors hash when the record is invalid.

So the conditional collapses. Three lines of body. Anything more, and I want to know why.

Jobs are controllers for async work

class Card::PublishJob < ApplicationJob
  def perform(card)
    card.publish
  end
end

ActionJob + GlobalID handle the lookup for you. The body collapses to the model call.

class Card::PublishWorker
  include Sidekiq::Worker
  include CardScoped

  def perform(card_id)
    scoped_card(card_id).publish
  end
end

Sidekiq passes raw IDs — the lookup belongs in a shared concern, just like in controllers.

ActionJob does a fantastic job of abstracting away the boilerplate from a job, leaving the single model call as the expected only line.

With Sidekiq there's more boilerplate to handle, so if it's necessary to use, id-to-instance lookups belong in a shared concern there too.

Chapter 3

Why this matters

Beyond aesthetics

← →

Read by shape, not by content

If every controller looks like the standard, then the interesting parts of any given action stand out. Where does it redirect? Is the rendering special? Everything else fades into convention.

Both these rules boil down to making sure model responsibilities are properly modeled, reusable, and part of the business domain.

It makes reading jobs and controllers obvious. Just by looking at the shape, the reader can extract that 'the standard is happening here.'

I might want to know where we redirect, or if there's anything special about the rendering — but that's it.

← →

Design from the domain, not the button

UI-first

"What happens when the user presses this button?"

  • A procedure tied to one click
  • Lives wherever the click was wired
  • Hard to reuse from a job, an API, or a test

Domain-first

"What behavior does our domain have?"

  • A method on the domain object
  • Lives next to the data it changes
  • Reusable from any controller, job, or console

From there, the controller designs itself. Not 'what JSON should we return for this click', but 'what controller represents this domain model.'

This also makes authors think about the domain in a way that's not coupled to the UI that initiated the request.

Instead of 'a procedure happening when a user presses a button,' it becomes 'what behavior does our domain have?'

Coming from that direction, the controller gets designed too. Instead of 'what JSON should we return when the user presses this button,' it becomes 'what controller represents this domain model.'

Takeaways

How I'd like us to think about it

  • Controllers orchestrate HTTP. Jobs orchestrate async work. Neither holds business logic.
  • The standard shape: authenticate → lookup → authorize → model call → response.
  • Share the shape with concerns — ApplicationController, Scoped.
  • Optimize for shape recognition, not cleverness.
  • Think domain-first, not UI-first.
  • Lean on RESTful actions to discover missing objects.
